10 # Coding
11 
12 QuEST can be used in your C or C++ code, simply by including
13 ```C
14 #include <QuEST.h>
15 ```
16 
17 Independent of which platform you'll run your simulation on (multicore CPUS, a GPU, or over a network), your QuEST code will look the same, compile with the same [makefile](https://github.com/TysonRayJones/QuEST/blob/master/makefile), and use the same [API](https://tysonrayjones.github.io/QuEST/QuEST_8h.html).
18 
19 Here's a simulation of a very simple circuit which measures ![equation](https://latex.codecogs.com/gif.latex?C_0%28X_1%29%20H_0%20%7C00%5Crangle).
20 ```C
21 #include <QuEST.h>
22 
23 int main(int narg, char *varg[]) {
24 
25  // load QuEST
26  QuESTEnv env;
27  initQuESTEnv(&env);
28 
29  // create 2 qubits in the zero state
30  MultiQubit qubits;
31  createMultiQubit(&qubits, 2, env);
32  initStateZero(qubits);
33 
34  // apply circuit
35  hadamard(qubits, 0);
36  controlledNot(qubits, 0, 1);
37  measure(qubits, 1);
38 
39  // unload QuEST
40  destroyMultiQubit(qubits, env);
41  closeQuESTEnv(env);
42  return 0;
43 }
44 ```
45 Of course, this code doesn't output anything!

49 
50 Let's walk through a more sophisticated circuit.
51 
52 We first construct a quest environment, which abstracts away any preparation of multithreading, distribution or GPU-acceleration strategies.
53 ```C
54 QuESTEnv env;
55 initQuESTEnv(&env);
56 ```
57 
58 We then create a quantum register, in this case of 3 qubits.
59 ```C
60 MultiQubit qubits;
61 createMultiQubit(&qubits, 3, env);
62 ```
63 and set it to be in the zero state.
64 ```C
65 initStateZero(qubits);
66 ```
67 We can create multiple `MultiQubit` instances, and QuEST will sort out allocating memory for the state-vectors, even over networks!
68 
69 We're now ready to apply some gates to our qubits, which in this case have indices 0, 1 and 2.
70 When applying a gate, we pass along which quantum register to operate upon.
71 ```C
72 hadamard(qubits, 0);
73 controlledNot(qubits, 0, 1);
74 rotateY(qubits, 2, .1);
75 ```
76 
77 Some gates allow us to specify a general number of control qubits
78 ```C
79 multiControlledPhaseGate(qubits, (int []){0, 1, 2}, 3);
80 ```
81 
82 We can specify general single-qubit unitary operations as 2x2 matrices
83 ```C
84 // sqrt(X) with a pi/4 global phase
85 ComplexMatrix2 u;
86 u.r0c0 = (Complex) {.real=.5, .imag= .5};
87 u.r0c1 = (Complex) {.real=.5, .imag=-.5};
88 u.r1c0 = (Complex) {.real=.5, .imag=-.5};
89 u.r1c1 = (Complex) {.real=.5, .imag= .5};
90 unitary(qubits, 0, u);
91 ```
92 or more compactly, foregoing the global phase factor,
93 ```C
94 Complex a, b;
95 a.real = .5; a.imag = .5;
96 b.real = .5; b.imag = -.5;
97 compactUnitary(qubits, 1, a, b);
98 ```
99 or even more compactly, as a rotation around an arbitrary axis on the Bloch-sphere
100 ```C
101 Vector v;
102 v.x = 1; v.y = 0; v.z = 0;
103 rotateAroundAxis(qubits, 2, 3.14/2, v);
104 ```
105 
106 We can controlled-apply general unitaries
107 ```C
108 controlledCompactUnitary(qubits, 0, 1, a, b);
109 ```
110 even with multiple control qubits!
111 ```C
112 multiControlledUnitary(qubits, (int []){0, 1}, 2, 2, u);
113 ```
114 
115 What has this done to the probability of the basis state |111> = |7>?
116 ```C
117 REAL prob = getProbEl(qubits, 7);
118 printf("Probability amplitude of |111>: %lf\n", prob);
119 ```
120 Here, `REAL` is a floating point number (e.g. `double`). The state-vector is stored as `REAL`s so that we can change its precision without any recoding, by configuring [QuEST_precision.h](../QuEST/QuEST_precision.h)
121 
122 How probable is measuring our final qubit (2) in outcome `1`?
123 ```C
124 prob = findProbabilityOfOutcome(qubits, 2, 1);
125 printf("Probability of qubit 2 being in state 1: %f\n", prob);
126 ```
127 
128 Let's measure the first qubit, randomly collapsing it to 0 or 1
129 ```C
130 int outcome = measure(qubits, 0);
131 printf("Qubit 0 was measured in state %d\n", outcome);
132 ```
133 and now measure our final qubit, while also learning of the probability of its outcome.
134 ```
135 outcome = measureWithStats(qubits, 2, &prob);
136 printf("Qubit 2 collapsed to %d with probability %f\n", outcome, prob);
137 ```
138 
139 At the conclusion of our circuit, we should free up the memory used by our state-vector.
140 ```C
141 destroyMultiQubit(qubits, env);
142 closeQuESTEnv(env);
143 return 0;
144 ```

150 and after compiling (see section below), gives psuedo-random output

152 > ```
153 > Probability amplitude of |111>: 0.498751
154 > Probability of qubit 2 being in state 1: 0.749178
155 > Qubit 0 was measured in state 1
156 > Qubit 2 collapsed to 1 with probability 0.998752
157 > ```
158 
159 > ```
160 > Probability amplitude of |111>: 0.498751
161 > Probability of qubit 2 being in state 1: 0.749178
162 > Qubit 0 was measured in state 0
163 > Qubit 2 collapsed to 1 with probability 0.499604
164 > ```
165 
166 QuEST uses the [Mersenne Twister](http://www.math.sci.hiroshima-u.ac.jp/~m-mat/MT/MT2002/emt19937ar.html) algorithm to generate random numbers used for randomly collapsing the state-vector. The user can seed this RNG using `seedQuEST(arrayOfSeeds, arrayLength)`, otherwise QuEST will by default (through `seedQuESTDefault()`) create a seed from the current time, the process id, and the hostname.

170 # Compiling
171 
172 To compile, copy the [makefile](../makefile) into the same folder as your circuit code. Adjust the *User Settings* section to configure compilation. You'll need to set
173 ```bash
174 # name of the executable to create
175 EXE = myExecutable
176 
177 # space-separated names (no file type) of all user source files (.c or .cpp) in the root directory
178 SOURCES = myCode1 myCode2
179 
180 # path to QuEST library from root directory
181 QUEST_DIR = path/to/QuEST
182 ```
183 
184 Next, indicate which compiler you wish to use. For example, to use the default compiler on OSX:
185 ```bash
186 # compiler to use, which should support both C and C++, to be wrapped by GPU/MPI compilers
187 COMPILER = clang
188 
189 # type of above compiler, one of {GNU, INTEL, CLANG}, used for setting compiler flags
190 COMPILER_TYPE = CLANG
191 ```
192 
193 To compile your code to run on multicore/multi-CPU systems, and/or for distributed systems, or on GPUs, simply set the appropriate variables
194 ```bash
195 # hardwares to target: 1 means use, 0 means don't use
196 MULTITHREADED = 0
197 DISTRIBUTED = 0
198 GPUACCELERATED = 0
199 ```
200 Note that using multithreading requires an OpenMP compatible compiler (e.g. [GCC 4.9](https://gcc.gnu.org/gcc-4.9/changes.html)), using distribution requires an MPI compiler (`mpicc`)is installed on your system, and GPU acceleration requires a CUDA compiler (`nvcc`). We've made a comprehensive list of compatible compilers which you can view [here](../tests/compilers/compatibility.md). This does not change your `COMPILER` setting - the makefile will choose the appropriate MPI and CUDA wrappers automatically.
201 
202 > Note also that GPU users must additionally specify the the *Compute Capability* of their GPU, which can be looked up at the [NVIDIA website](https://developer.nvidia.com/cuda-gpus)
203 > ```bash
204 > GPU_COMPUTE_CAPABILITY = 30
205 > ```
206 > An incorrect *Compute Capability* will lead to drastically incorrect computations. You can check if you've set the right *Compute Capability* by running the unit tests via `tests/runTests.c`.
207 
208 You're now ready to compile your code by entering
209 ```bash
210 make
211 ```
212 at the terminal, in the directory of your code. For the above example, this performs
213 ```bash
214 gcc -O2 -std=c99 -mavx -Wall -fopenmp -c path/to/QuEST/CPU/QuEST.c
215 gcc -O2 -std=c99 -mavx -Wall -fopenmp -c path/to/QuEST/mt19937ar.c
216 gcc -O2 -std=c99 -mavx -Wall -fopenmp -c myCode1.c
217 gcc -O2 -std=c99 -mavx -Wall -fopenmp -c myCode2.c
218 gcc -O2 -std=c99 -mavx -Wall -fopenmp -c path/to/QuEST/CPU/QuEST_env_local.c
219 gcc -O2 -std=c99 -mavx -Wall -fopenmp -o myExecutable QuEST.o mt19937ar.o myCode1.o myCode2.o QuEST_env_local.o -lm
220 ```

224 # Running
225 
226 ## locally
227 
228 You can then call your code
229 ```bash
230 ./myExecutable
231 ```
232 If you enabled multithreading when compiling, you can control how many threads your code uses by setting `OMP_NUM_THREADS`, ideally to the number of available cores on your machine
233 ```bash
234 export OMP_NUM_THREADS=8
235 ./myExecutable
236 ```
237 QuEST will automatically allocate work between the given number of threads to speedup your simulation.
238 
239 If you compiled in distributed mode, your code can be run over a network (here, over 8 machines) using
240 ```bash
241 mpirun -np 8 ./myExecutable
242 ```
243 This will, if enabled, also utilise multithreading on each node with as many threads set in `OMP_NUM_THREADS`.
244 
245 If you compiled for a GPU connected to your system, simply run
246 ```bash
247 ./myExecutable
248 ```
249 as normal!
250 
251 ## through a job submission system
252 
253 There are no special requirements for running QuEST through job submission systems. Just call `./myExecutable` as you would any other binary.
254 
255 For example, the [above code](tutorial_example.c) can be split over 4 MPI nodes (each with 8 cores) by setting `DISTRIBUTED = 1` (and `MULTITHREADED = 1`) in the makefile, and writing a SLURM submission script
256 ```bash
257 #SBATCH --nodes=4
258 #SBATCH --ntasks-per-node=1
259 
260 module purge
261 module load mvapich2
262 
263 make clean
264 make
265 
266 export OMP_NUM_THREADS=8
267 mpirun ./myExecutable
268 ```
269 or a PBS submission script like
270 ```bash
271 #PBS -l select=4:ncpus=8
272 
273 make clean
274 make
275 
276 export OMP_NUM_THREADS=8
277 aprun -n 4 -d 8 -cc numa_node ./myExecutable
278 ```
279 
280 Running QuEST on a GPU partition is similarly easy in SLURM
281 ```bash
282 #SBATCH --nodes=1
283 #SBATCH --ntasks-per-node=1
284 #SBATCH --gres=gpu:1
285 
286 #SBATCH --partition=gpu ## name may vary
287 
288 module purge
289 module load cuda ## name may vary
290 
291 make clean
292 make
293 
294 ./myExecutable
295 ```
296 
297 On each platform, there is no change to our source code or our QuEST interface. We simply recompile, and QuEST will utilise the available hardware (a GPU, shared-memory or distributed CPUs) to speedup our code.
298 
299 Note that parallelising with MPI (`DISTRIBUTED = 1`) will mean all code in your source file will be repeated on every node. To execute some code (e.g. printing) only on one node, do
300 ```C
301 if (env.rank == 0)
302  printf("Only one node executes this print!");
303 ```
304 Such conditions are valid and always satisfied in code run on a single node.
